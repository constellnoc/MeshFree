# MeshFree 产品需求池

## 1. 文档目的

本文档用于记录 `MeshFree` 当前阶段已经识别出的产品问题、体验改进项和后续扩展方向。

它不是当前版本必须一次性全部完成的开发清单，而是一个持续维护的产品需求池，用来帮助后续：

- 统一问题表述
- 明确优先级
- 控制 MVP 范围
- 支持后续迭代规划

## 2. 当前项目阶段

当前项目处于：

- `MeshFree MVP` 三条核心业务闭环已打通
- 真实服务器部署已完成
- 当前重点从“先把功能做出来”转向“继续验证、补关键体验问题、按需做小修复和结构整理”

因此，本需求池中的事项默认遵循以下原则：

- 优先处理会影响当前可用性和风险的问题
- 谨慎推进结构性优化
- 暂不优先实现明显超出当前 MVP 范围的大功能

## 3. 优先级说明

### `P0`

当前阶段最值得优先推进的问题。

这些问题通常会直接影响：

- 实际可用性
- 用户输入安全
- 资源提交风险
- 网站主流程清晰度

### `P1`

近期适合讨论并逐步落地的问题。

这些问题通常会提升：

- 产品感
- 信息结构
- 导航清晰度
- 后台使用体验

### `P2`

适合保留、继续讨论，但当前不宜优先投入的问题。

这些问题通常属于：

- 结构性升级
- 长期扩展方向
- 超出当前 MVP 范围的能力

## 4. 当前阶段短版路线图

9. 管理员权限提示与危险操作保护

14. 网站资源接入 `UE` 的导入能力

## 5. 正式 Backlog

### 5.1 表单草稿保护

- 优先级：`P0`
- 当前判断：应优先讨论并尽快解决
- 核心目标：避免用户在投稿或输入过程中因切换页面而丢失已填写内容
- 关键点：
  - 哪些页面存在输入丢失风险
  - 是否需要离开提醒
  - 是否需要本地临时保存
- 建议推进方式：
  1. 先锁定投稿页为首要保护对象
  2. 设计草稿保存与清理规则
  3. 后续再评估后台输入区是否也要接入

### 5.2 ZIP 上传前预览与校验信息

- 优先级：`P0`
- 当前判断：应优先讨论，较适合近期落地
- 核心目标：降低用户上传错误 ZIP 的风险，提高资源提交质量
- 关键点：
  - 是否只显示文件名、大小、格式
  - 是否需要解析 ZIP 内部目录结构
  - 前端预览和后端校验的边界
- 建议推进方式：
  1. 先做基础信息预览
  2. 再评估是否增加 ZIP 内部文件列表
  3. 保持实现轻量，不超出 MVP

### 5.3 移除管理员登录页默认用户名

- 优先级：`P0`
- 当前判断：应尽快处理
- 核心目标：去掉开发测试痕迹，提升产品专业度
- 关键点：
  - 默认值与 placeholder 的区别
  - 是否公开暗示管理员账号名
- 建议推进方式：
  1. 删除默认值
  2. 使用普通提示文案
  3. 保持登录页更中性

### 5.4 管理员统一入口逻辑

- 优先级：`P0`
- 当前判断：值得尽早定规则
- 核心目标：将管理员登录与后台访问整合为更一致的产品入口
- 关键点：
  - 已登录如何直达 `dashboard`
  - 未登录如何跳转到登录页
  - 退出后的回退路径
- 建议推进方式：
  1. 页面上只保留一个管理员入口
  2. 入口按登录态自动分流
  3. 页面职责仍保持“登录页 / dashboard”分离

### 5.5 顶部 `tabbar` 结构重设计

- 优先级：`P0`
- 当前判断：适合现在先定方向
- 核心目标：让顶部导航更符合游客主流程和产品主路径
- 关键点：
  - 顶部哪些项必须保留
  - 管理员入口是否应下沉
  - 为未来普通用户功能预留空间
- 建议推进方式：
  1. 以游客核心路径为中心重排导航
  2. 避免后台功能长期占主导航重点
  3. 与 `footer` 一起统一考虑

### 5.6 Footer 信息区建设

- 优先级：`P1`
- 当前判断：适合近期落地
- 核心目标：补足站点的基础说明区，提升完整度与可信度
- 关键点：
  - About 放什么内容
  - GitHub 和作者信息如何呈现
  - 是否与管理员入口合并布局
- 建议推进方式：
  1. 增加统一 `footer` 区块
  2. 放入 About、GitHub、作者信息
  3. 保持信息简洁，不做重型内容页

### 5.7 管理员入口下沉到底部

- 优先级：`P1`
- 当前判断：方向合理，应和导航重构一起考虑
- 核心目标：弱化后台入口在公开页面中的视觉权重
- 关键点：
  - 是否仍需保留访问便利性
  - 与 `footer` 信息区如何共存
- 建议推进方式：
  1. 管理员入口从顶部主导航移出
  2. 在 `footer` 中作为次级入口展示
  3. 与统一入口逻辑联动

### 5.8 图标、标题与站点命名统一

- 优先级：`P1`
- 当前判断：适合较早处理
- 核心目标：统一站点品牌展示，提高演示和浏览器层面的辨识度
- 关键点：
  - 正式站名
  - 浏览器标题规范
  - `favicon` 与后台页面命名
- 建议推进方式：
  1. 明确站点主名称
  2. 统一页面 `title` 格式
  3. 补 `favicon` 与基础品牌信息

### 5.9 管理员权限提示与危险操作保护

- 优先级：`P1`
- 当前判断：值得近期定规范
- 核心目标：让后台操作更有权限感，并降低误操作风险
- 关键点：
  - 哪些操作需要二次确认
  - 是否需要后台专属视觉标识
  - 危险按钮如何表现
- 建议推进方式：
  1. 优先补危险操作保护
  2. 再补后台视觉提示
  3. 不急着做完全不同主题

### 5.10 页面打开规则统一

- 优先级：`P1`
- 当前判断：适合现在先立规矩
- 核心目标：统一站内、站外和下载行为的打开逻辑，减少交互混乱
- 关键点：
  - 站内页面是否一律当前页
  - 站外链接是否一律新标签
  - 下载行为是否保持当前页
- 建议推进方式：
  1. 先制定全站统一规则
  2. 后续按规则检查各入口
  3. 特殊例外单独记录

### 5.11 中英文界面切换

- 优先级：`P2`
- 当前判断：方向合理，但不建议现在零散上
- 核心目标：提供中英文界面切换能力，提升展示性和国际化完整度
- 关键点：
  - 文案是否统一抽离
  - 错误提示与按钮如何双语维护
  - 后续新增页面如何同步
- 建议推进方式：
  1. 先决定是否列入下一阶段重点
  2. 如果做，就按完整国际化机制推进
  3. 避免临时拼接式实现

### 5.12 UI / performance 问题拆分

- 优先级：`P2`
- 当前判断：不适合作为笼统任务直接推进
- 核心目标：把模糊的“界面性能问题”拆成可判断、可执行的具体问题
- 关键点：
  - 是视觉问题还是性能问题
  - 是否存在真实卡顿，还是只是体验粗糙
  - 优化是否有明确指标
- 建议推进方式：
  1. 先拆问题
  2. 再按类别进入 backlog
  3. 避免把泛化概念直接当任务做

### 5.13 普通用户与管理员账户体系关系

- 优先级：`P2`
- 当前判断：适合先定长期方向，不适合现在重构
- 核心目标：为未来用户系统扩展提前明确权限模型
- 关键点：
  - 管理员是否属于同一用户体系
  - 权限差异如何表达
  - 当前 MVP 是否需要为未来预埋结构
- 建议推进方式：
  1. 当前保留独立管理员实现
  2. 长期方向定义为“同一账户体系 + 角色权限”
  3. 等普通用户功能进入范围后再重构

### 5.14 网站资源接入 `UE` 的导入能力

- 优先级：`P2`
- 当前判断：值得保留为方向，但不急于实现
- 核心目标：提升网站资源导入 `UE` 的便利性，探索从资源站走向工具链平台的可能
- 关键点：
  - 是“更方便导入”还是“深度接入 UE 工作流”
  - 是资源规范问题、工具问题，还是插件问题
  - 是否属于当前项目阶段范围
- 建议推进方式：
  1. 先作为长期扩展方向保留
  2. 短期先考虑 UE 友好资源规范
  3. 长期再考虑插件或桥接工具

## 6. 推荐近期聚焦范围

如果要从当前需求池中压缩出一个更适合近期推进的小范围任务集合，建议优先关注：

- 表单草稿保护
- ZIP 上传前预览与校验信息
- 移除管理员登录页默认用户名
- 管理员统一入口逻辑
- 顶部 `tabbar` 结构重设计
- Footer 信息区建设与管理员入口下沉

这些事项最符合当前阶段“在不明显扩展范围的前提下，优先提升真实体验、结构清晰度和产品感”的目标。
