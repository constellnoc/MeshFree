# MeshFree 启动提示词

## 1. 文档角色

- `README.md`：GitHub 对外介绍。优先写项目介绍、公开能力、结构入口、文档入口。
- `docs/README.md`：内部文档入口。负责说明文档体系、读取顺序、去哪里找资料。
- `docs/session-notes/prompt.md`：协作契约。只保留长期稳定、跨任务通用的规则。
- 正式文档：稳定规则。包括产品范围、API 契约、数据结构、工作流、部署与维护流程。
- `docs/session-notes/`：变化记录。保存每轮推进、临时决策、联调排障、阶段结论。
判断规则：
- 会频繁变化、带时间味道 -> `session-notes`
- 需要长期复用、跨任务稳定 -> 正式文档
- 只和“怎么开始、怎么协作”有关 -> `prompt`

## 2. 权威模型

`prompt` 对以下内容有最高约束力：
- 流程
- 信任顺序
- 核验顺序
- 停手机制
- 文档治理纪律
- 沟通方式
`prompt` 不直接充当以下事实来源：
- 当前版本
- 当前能力
- 当前状态

用户本轮表述与 `prompt` 是并列高优先级。
如果用户表述与 `prompt` 冲突：
- 不得自行裁决
- 必须先报告冲突
- 等待用户确认
在 `prompt` 不覆盖的事实范围，默认优先参考：
1. 用户本轮明确指定
2. remote tag / remote release / remote refs
3. 已同步更新的 `CHANGELOG.md` + `docs/releases/*`
4. 最近相关 `session-notes`
补充：
- 这些只是优先参考，不是绝对真理
- 一旦与已核验事实明显冲突，必须报告
- 旧文案只能当线索，不能直接复用成当前事实

## 3. 内部默认读取顺序

默认读取顺序：
1. 先读本文件 `docs/session-notes/prompt.md`
2. 再读 `docs/README.md`
3. 如果任务涉及版本 / 发布 / 对外口径，默认加读 `CHANGELOG.md`
4. 按任务类型选读正式文档：
   - 产品 / 范围 / 当前能力：`docs/mvp-spec.zh-CN.md`
   - 接口 / 数据结构：`docs/api-design.zh-CN.md`
   - 多格式上传 / 转换 / 预览：`docs/multi-format-preview-workflow.zh-CN.md`
   - 部署 / 服务器 / HTTPS / Nginx / PM2：`docs/deployment-guide.zh-CN.md`
   - 部署执行清单：`docs/deployment-checklist.zh-CN.md`
   - 日常服务器维护：`docs/server-maintenance.zh-CN.md`
5. 再看最近且最相关的 `docs/session-notes/`
6. 只有在需要 GitHub 首页视角、对外介绍口径、仓库公开展示信息时，再读根 `README.md`
初始化开始项目时，`session-notes` 默认参考近期的就够了。
不要默认串读整段历史。只有在以下情况才回查更早记录：
- 用户明确要求引用旧记录
- 当前任务必须追溯历史决策
- 现有代码与正式文档冲突，需要核对演变过程
即使回查，也只查最相关的少数几份，不要整包重读。

## 4. `session-notes` 用法

默认原则：
- 一条 note 只服务一个明确主题
- 不要把多条无关主线强塞进同一篇
- 优先记录：本轮做什么、为什么做、改了哪里、验证到哪里、下一步是什么
建议文件名：
- `YYYYMMDD_n_topic.md`
建议结构：1. 本轮主题 2. 本轮目标 3. 已做改动 4. 关键结论 / 决策 5. 验证情况 6. 当前遗留点 7. 下一步建议
使用原则：
- 新任务先看最近且相关的 notes
- 不把 `session-notes` 当正式规格
- 某个结论稳定后，提炼回正式文档

## 5. 当前口径与核验规则

凡涉及以下表述，必须先核验再写：
- 当前版本
- 当前 release
- 当前状态
- 当前能力
- 当前 milestone
- 最新
- 已完成
- 已同步

总规则：
- 旧文案不是事实
- 未核验事实不得写成肯定句
- 未知就写未知
- 冲突先报告，不准顺手统一
- 不准凭旧文案续写当前口径

定义：
- 当前版本：默认指最近有效发布口径；优先看用户指定，再看 remote tag / release
- 当前能力：默认指当前代码 / 主分支已具备的能力
- 当前状态：默认指当前仓库和文档能核准的项目状态，不能只靠旧文案

版本核验规则：
只要文档里要写“当前版本 / 当前 milestone / 当前 release”，必须先核对：
1. 用户本轮说明
2. remote tag / remote refs / remote release
3. `CHANGELOG.md`
4. `docs/releases/*`
5. 最近相关 `session-notes`

若这些不一致：
- 先汇报冲突
- 不要继续写版本口径

如果 remote tag / release 已存在，但 `CHANGELOG.md` 或 `docs/releases/*` 未同步，默认先停下并询问用户。

证据摘要模式默认是 `risk-only-show`：
- 只在涉及版本 / release / 当前状态 / 当前能力
- 或发现冲突时
- 才显式展示证据摘要

证据摘要至少应覆盖：
- 用户口径
- remote tag / refs / release
- `CHANGELOG.md`
- `docs/releases/*`
- 最近相关 `session-notes`
- 必要时代码 / 运行结果
- 冲突与本次采用口径

## 6. 冲突与停手机制

一旦发现版本、能力、范围、流程文档互相冲突：
- 先列冲突
- 先核验
- 再告知用户

不允许凭感觉统一口径。
当用户口径与已核验事实冲突时，以下场景默认必须停下并等用户确认：
- 修改 `README.md`
- 修改 `CHANGELOG.md`
- 修改 `docs/releases/*`
- 修改正式文档里的“当前版本 / 当前状态 / 当前能力”表述
- 对外口径、面向他人的项目说明、公开介绍
普通讨论或计划阶段，如用户口径与已核验事实冲突：
- 可以继续讨论
- 但必须说明这是按用户口径暂定
- 不等于已核验事实

## 7. 文档更新纪律

文档如果保留，就要规范维护；不维护还不如不要假装它是当前状态。
正式文档更新规则：
- 产品范围、接口行为、数据结构、上传 / 下载 / 预览 / 审核工作流、部署步骤、维护步骤发生实质变化时，同轮同步更新对应正式文档
不要出现：
- 代码已经变了，正式文档还写旧规则
- `prompt` 写着最新状态，但正式文档没更新
- 把本应进正式文档的稳定结论长期埋在 `session-notes`
`CHANGELOG.md` 与 `docs/releases/*` 规则：
- 它们不是日记，是发布层文档
- 进入明确版本节点、里程碑节点、对外可见范围变化时，要同步更新
- 如果本轮已经在声明“当前版本 / 当前发布说明 / 当前变更记录”，就必须让这些文件同步
- 如果本轮没有维护这些文件，不要在别处把它们说成已经反映当前状态
- 不要长期保留明显过期的“当前 release”“当前 changelog”
- 如果没有先核准 release 文档、`CHANGELOG.md`、版本标签是否同步，不要在 `README.md`、`docs/README.md`、提示说明里硬写“当前版本是 X”
发布层文档最少要求：
- 版本摘要
- Added / Changed 或等价结构
- 验证状态
- 已知缺口
- 相关文档入口
重要对外文档默认按更严格口径维护：
- `README.md`
- `CHANGELOG.md`
- `docs/releases/*`
- `docs/api-design.en.md`
- `docs/deployment-guide.en.md`
规则：
- 不能长期过期
- 版本口径与对外口径更严格
- 若本轮涉及这些文档的当前口径更新，要同步检查相关发布层文档
任何会改变对外或对内“当前口径”的修改，都要检查是否同步影响：
- `README.md`
- `docs/README.md`
- `CHANGELOG.md`
- `docs/releases/*`
- 相关正式文档
- 相关 `session-notes`
如果本轮不维护这些文件：
- 不要写“当前已同步”
- 不要把它们写成已反映当前状态
双语规则：
- 不是所有文档都必须双语同步
- 默认更严格要求双语或对外同步的，是重要对外文档中的相关文件
- 例如：`docs/api-design.en.md`、`docs/deployment-guide.en.md`、`docs/releases/*`

## 8. 沟通规则

- 和我解释一律中文
- 代码注释用英文
- 说明顺序默认：先讲是什么，再讲为什么，再讲怎么做
- 我是前后端新手，默认不要跳步
- 开始编码前，先说本轮要做什么、为什么这样做
- 准备改文件前，先说会改什么
- 准备跑长命令前，先说目的
- 每次回复收尾时，都要明确建议用户下一步该怎么做，并询问是否按这个方向继续

## 9. 行为边界

- 不要擅自扩范围
- 除非我明确要求，不要提交 git commit
- 如果适合给 commit message，只给建议，不要代提交
- 不要把敏感信息写进仓库
不要提交：
- `.env`
- 管理员真实密码
- JWT 密钥
- 数据库文件
- 上传文件

## 10. 部署相关规则

如果任务涉及：

- 服务器联调
- 域名
- HTTPS
- Nginx
- PM2
- 环境变量
- 生产数据库
则优先沿用现有部署文档，不要另起一套。
相关入口：
- `docs/deployment-guide.zh-CN.md`
- `docs/deployment-guide.en.md`
- `docs/deployment-checklist.zh-CN.md`
- `docs/server-maintenance.zh-CN.md`

## 11. 每轮开始时默认输出

新一轮开始时，先短答这 3 件事：

1. 当前项目大致处于什么阶段
2. 本轮直接相关上下文来自哪些文档 / 最近 notes
3. 本轮准备做什么
补充：
- 默认先引用正式文档和最近相关 `session-notes`
- 不要为了“显得全面”去重读大量旧记录
