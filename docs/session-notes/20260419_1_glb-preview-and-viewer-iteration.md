# MeshFree ZIP 内 GLB 预览与 Viewer 迭代总结

## 1. 本轮主题

本轮工作的主题，是把之前还停留在“未来方向”的真实 3D 预览，正式推进到可运行的第一版实现。

这轮不再只是讨论“是否要做 viewer”，而是实际围绕以下问题展开并落地：

- 投稿资源是否继续统一使用 `ZIP`
- 如何从 `ZIP` 中自动识别可预览的 `.glb`
- 如何在前端详情交互中接入真实 3D 预览
- 如何根据真实样例模型继续迭代 viewer 的交互和观感

截至本轮结束，项目重点已经从“弹窗壳子已搭好”推进到“真实 3D 预览链路已初步打通，并开始进入 viewer 体验打磨阶段”。

## 2. 本轮已完成内容

### 2.1 已确认并落地 `ZIP -> 自动识别 GLB -> 详情预览` 路线

本轮已经正式确认并按此实现：

- 投稿入口继续保持 `cover + zip`
- 不额外新增“单独上传 glb”分支
- 服务端会在上传成功后自动尝试从 `ZIP` 中寻找第一个 `.glb`
- 找到后保存预览模型路径
- 找不到时不阻断投稿主流程

这意味着当前项目已经从“只保存封面图和 ZIP 下载”推进到“开始具备真实预览模型抽取能力”。

### 2.2 数据库与后端预览链路已打通

本轮已在后端完成：

- 为 `Submission` 新增 `previewModelPath`
- 新增独立工具模块处理 `ZIP` 中的 `.glb` 抽取
- 详情接口返回 `previewModelUrl`
- 删除投稿时同步清理已抽取的预览模型文件

这意味着当前预览能力不是前端假数据演示，而是已经接入到真实数据结构和上传链路中。

### 2.3 详情页已支持从封面图进入真实 3D 预览

本轮已将模型详情的左侧图片区改为：

- 默认先显示封面图
- 悬停时显示 `Preview`
- 点击后打开独立的大 viewer 弹层

而不是：

- 在原图片区内直接长期自动加载 3D
- 或一开始就把 viewer 强行嵌到详情布局里

这样做的结果是：

- 首屏仍然保持轻量
- 用户操作意图更明确
- viewer 能作为一个真正的“大预览框”存在，而不是图片区的小替换块

### 2.4 Viewer 大弹层已完成第一版交互

本轮 viewer 已具备：

- 点击 `Preview` 后打开独立大预览框
- 鼠标拖动旋转
- 滚轮缩放
- `Reset view`
- `Close`
- 支持 `Esc` 关闭

同时，viewer 的 reset 逻辑也已从“粗暴重建画布”调整为“真正复位相机和 controls”，避免了缩放异常和突然跳变的问题。

### 2.5 已围绕真实样例模型继续修正材质与控制问题

本轮不是只停留在“viewer 能跑”，还实际用桌面的真实样例 `test_c.zip` 做了验证和调整。

围绕这份真实样例，已完成：

- 在本地库中生成一条已审核的测试模型
- 检查 `GLB` 材质结构
- 确认该模型使用了 `KHR_materials_pbrSpecularGlossiness`
- 补上这类旧材质的兼容处理，尝试恢复 `diffuseTexture` / `normalTexture`
- 根据实际观感继续压低过油、过亮的表面质感

这意味着当前 viewer 已经不只是“对普通新式 glTF 可能可用”，而是已经开始处理真实模型兼容问题。

### 2.6 Viewer 背景已支持四个预设色

根据后续反馈，本轮又继续补了背景预设，并改为默认白底。

当前支持四档：

- `White`
- `Black`
- `Soft White`
- `Soft Black`

并且：

- 默认背景改为 `White`
- viewer 右上角可直接切换
- 按钮与提示文字会随背景风格做基础适配

这使得当前 viewer 已经开始具备“切换不同展示环境观察模型”的基础能力。

## 4. 当前已确认的实现细节

### 4.1 第一版预览文件规则

当前已明确：

- 第一版只识别 `ZIP` 中的 `.glb`
- 暂不处理 `.gltf + 资源文件`
- 如果 `ZIP` 中没有 `.glb`，则保持封面图和 ZIP 下载体验

这意味着当前预览能力仍然是严格收敛在 MVP 范围内的。

### 4.2 Viewer 的当前交互结构

当前交互已改为：

1. 在详情里先看封面图
2. 悬停出现 `Preview`
3. 点击后打开独立 viewer modal
4. 在 modal 中进行旋转、缩放、reset 和关闭

这意味着当前 viewer 的职责已经和详情信息区分开，不再只是详情布局里的一个“图片区替换块”。

### 4.3 当前材质兼容仍在继续打磨

虽然本轮已经补了对 `KHR_materials_pbrSpecularGlossiness` 的第一轮兼容，但当前仍然不能认为材质观感已经完全达到目标。

当前已知结论包括：

- 颜色缺失问题不只是灯光问题
- 真实原因与旧材质扩展兼容有关
- 当前 viewer 已开始手动恢复贴图和法线图
- 但“像真实花瓣材质”的最终效果仍需继续精调

也就是说，当前 viewer 已从“完全白模/错材质”推进到“开始进入真实材质调校阶段”，但还没有完全收口。

## 5. 本轮验证与联调情况

本轮已做的验证包括：

- 前后端构建检查
- 使用包含 `.glb` 的 ZIP 验证抽取结果
- 验证无 `.glb` 时 `previewModelUrl` 为空
- 使用桌面真实样例 `test_c.zip` 走本地 UI 联调
- 确认本地测试模型详情可返回 `previewModelUrl`

本轮也遇到并处理了若干本地联调问题，例如：

- `better-sqlite3` 与当前 Node 版本不匹配
- 旧后端进程占用端口导致接口返回旧结果
- 本地 viewer 体验需要基于真实样例不断调整

这意味着当前这条链路已经不是纸面计划，而是经过了实际本地运行验证。

## 6. 当前仍可继续优化的点

本轮虽然已经完成第一版真实预览链路，但以下内容仍然存在明显优化空间：

- 旧材质扩展模型的材质真实感仍需继续收敛
- viewer 的默认镜头与 framing 还可以更像专业预览器
- 背景切换按钮还可继续增强可理解性
- 背景切换后的灯光观感还可以进一步按背景联动优化
- 未来如需支持更多格式，还需要继续扩展 ZIP 内识别规则

目前这些问题已经不再是“有没有 viewer”，而是“viewer 是否足够好用、足够像真实预览器”。

## 7. 当前阶段结论

截至 `20260419_1`，项目已经具备：

- 统一 `ZIP` 投稿的第一版 `GLB` 自动识别链路
- 详情接口返回真实 `previewModelUrl`
- 默认封面图 + 点击进入 viewer 的详情交互
- 独立的大型 3D 预览弹层
- 基础的旋转、缩放、reset、关闭能力
- 围绕真实样例模型进行材质兼容和观感调整
- 四档 viewer 背景预设，且默认白底

这意味着当前前端已经从：

- 只有封面图和 ZIP 下载的详情展示

推进到：

- 开始具备真实 3D 预览能力、并进入 viewer 体验打磨阶段的模型浏览体验

## 8. 下一步建议

下一步建议优先考虑以下方向：

- 继续优先打磨 `KHR_materials_pbrSpecularGlossiness` 这类旧材质模型的真实观感
- 继续细化 viewer 的默认镜头、背景与灯光关系
- 如需扩格式，优先考虑 `ZIP` 内 `gltf + 资源文件` 的支持
- 若后续 viewer 功能继续变复杂，可考虑将控制条、背景切换和状态提示进一步组件化
